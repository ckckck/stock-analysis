# 全局文字缩放快捷键设计

## 背景

当前桌面端没有统一的页面级字体缩放能力。用户希望在主分支上直接支持 `cmd/ctrl +` 与 `cmd/ctrl -`，让页面整体的文字和图标随之变大/变小，并在下次启动时恢复到上次缩放值。

## 方案选择

### 方案 1：调整根字号

- 监听全局键盘事件，拦截 `cmd/ctrl +/-`
- 把缩放值映射为根节点 `html` 的 `font-size`
- 绝大多数基于 `rem` 的 Tailwind 文本、间距和 Lucide 图标会一起缩放
- 把缩放值持久化到现有 `config.layout`

这是本次采用的方案。改动集中，和现有主题变量注入方式兼容，也能沿用当前配置落盘链路。

### 方案 2：外层容器 transform 缩放

- 视觉上更接近“整页缩放”
- 但会影响滚动、点击区域、拖拽和布局测量

不采用。

### 方案 3：逐组件接入缩放变量

- 可控性高
- 改动面过大，不适合这次需求

不采用。

## 设计

### 配置层

- 在 `LayoutConfig` 中新增 `textScalePercent`
- 默认值为 `100`
- 旧配置文件若缺少该字段，后端加载时自动补默认值

### 前端行为

- 启动时从 `config.layout.textScalePercent` 读取缩放值
- 通过 `document.documentElement.style.fontSize = "<percent>%"` 应用全局缩放
- 拦截：
  - `cmd/ctrl +`
  - `cmd/ctrl =`
  - `cmd/ctrl -`
  - 数字键盘加减号
- 每次步进调整 `10%`
- 范围限制在 `80%` 到 `140%`
- 调整时阻止 WebView 默认缩放行为
- 与现有布局保存共用同一套 `saveLayoutConfig` 防抖保存逻辑

### 测试

- 前端纯函数测试：
  - 快捷键识别
  - 步进逻辑
  - 边界限制
- 后端配置测试：
  - 旧配置缺少 `textScalePercent` 时自动补成 `100`
