# AI 筛选同步长期无新增股票排除设计

## 背景

当前 AI 筛选同步的股票池来自编译时嵌入的 `stock_basic.json`。同步入口只会过滤 `IsActive == false` 的股票，但 `IsActive` 又完全依赖基础库里的 `list_status != "D"`。结果是部分已经明显无法继续补到最新交易日的股票，仍会长期留在同步候选中。

这类股票在本地库中的 `latest_bar` 明显早于目标交易日，但日线数据源返回的结果最后日期也同样停在较早时间。同步链路因此每次都会把它们判定为“缺少目标交易日”，每次重新入队，又每次在 `filterBarsForSync()` 后得到空集，最终持续出现 `no new bars`。

## 目标

1. 这类长期无新增日线的股票不再反复进入后续同步队列。
2. 顶部同步覆盖率 `(已同步/总数)` 不再被这类股票永久拖住。
3. 不污染 `stocks_basic.is_active` 的原始语义，避免把“基础股票名录状态”和“本地同步资格”混在一起。
4. 若未来上游数据恢复，系统可以自动重新尝试，而不是永久拉黑。

## 方案比较

### 方案 A：继续只依赖 `is_active`

- 优点：无需新增状态表。
- 缺点：当前问题已经证明基础股票名录和日线源可用性并不一致，无法解决反复 `no new bars`。

### 方案 B：直接改写 `stocks_basic.is_active`

- 优点：改动面小，很多现有查询天然会跟着过滤。
- 缺点：把“上游是否活跃”和“本地是否应继续同步”混成一个字段，后续难恢复，也容易误导筛选语义。

### 方案 C：新增同步排除状态表

- 优点：把“基础资料状态”和“同步资格状态”分开；可记录连续无新增次数、最近源日期、排除原因；同步队列和覆盖率统计都能统一复用。
- 缺点：需要扩展 SQLite schema 和几处 store/service 查询。

推荐方案：C。

## 设计

### 1. 新增同步排除状态表

新增一张本地表，按 `symbol` 记录：

- `no_new_streak`
- `last_local_trade_date`
- `last_source_trade_date`
- `last_target_trade_date`
- `excluded`
- `exclude_reason`
- `updated_at`
- `excluded_at`

这张表只表达“本地同步资格”，不替代基础股票信息。

### 2. 排除触发条件

当某只股票满足以下条件时，标记为同步排除：

- 连续 3 次同步结果都是 `no new bars`
- 且本次数据源返回的最新日期仍早于目标交易日
- 且两者之间至少落后 20 个交易日

其中“20 个交易日”通过交易日列表计算，而不是自然日。

### 3. 状态更新规则

- 成功写入新 bars：清空该股票的无新增计数，并取消排除状态。
- 单次 `no new bars` 但不满足阈值：只累计计数，不排除。
- 取数失败：不累计到 `no_new_streak`，因为这属于源异常，不是“源已无新数据”。
- 已被排除的股票：默认不再进入同步队列，也不计入同步覆盖率总数。

### 4. 覆盖率与筛选范围

同步服务中的两处入口都要复用同一套排除逻辑：

- 真正的同步候选构建
- `attachCoverageSummary()` 的 `MarketStockCount / PendingSyncStocks / SyncedToLatestStocks`

这样顶栏状态和实际同步队列才会一致。

### 5. 自动恢复

如果某只股票已在排除名单中，但之后基础股票池重新出现且本地/远端状态变化，可以在手动同步时重新尝试一次；一旦成功写入新 bars，就自动解除排除。

本次先实现“成功后自动解除”，不额外做复杂的定时重试策略。

## 测试

需要覆盖：

1. 连续 3 次 `no new bars` 且交易日滞后达到阈值后，该股票进入排除名单。
2. 被排除的股票后续同步不再调用日线取数。
3. 覆盖率统计会忽略已排除股票。
4. 单票取数失败不会错误累计到“长期无新增”。
5. 一旦后续拿到新 bars，会清空排除状态并恢复正常同步资格。
