# 顶部同步入口与通用同步弹窗设计

## 背景

当前顶部栏缺少独立的数据同步入口，用户只能在 AI 筛选流程里被动进入同步确认。设置面板里仍保留“软件更新”菜单项，但这一项不再是当前工作流中的核心操作。与此同时，筛选确认弹窗和同步进度弹窗头部文案区域没有稳定的左对齐，影响信息扫描效率。后续验证还发现两个交互问题：历史筛选恢复到当前结果后，用户即使修改了筛选文字，重新执行时仍可能错误复用历史 SQL；顶部同步按钮在取消后没有即时体现断点进度，再次同步时视觉上看起来像从 0 开始。

## 目标

1. 移除设置面板中的“软件更新”菜单项。
2. 在顶部栏“龙虎榜”前新增同步状态按钮，直接展示同步完成度。
3. 将现有“同步后开始筛选”弹窗抽象成通用同步弹窗，支持“仅同步”和“同步后筛选”两种模式。
4. 调整两个弹窗头部文案区域为左对齐。
5. 让历史筛选结果在“未改输入”时明确提示“根据历史筛选方式重新筛选”，并只在该条件下复用历史 SQL。
6. 让顶部同步按钮在取消后即时体现已同步数量和断点续传状态。

## 方案

### 顶部同步按钮

- 位置放在顶部栏“龙虎榜”按钮前，保持与现有文字按钮一致的圆角卡片风格。
- 按钮主文案与状态规则：
  - 全量已同步：显示“已同步”，绿色，不可点击。
  - 部分已同步：显示“立即同步”，黄色，可点击。
  - 完全未同步：显示“立即同步”，红色，可点击。
  - 同步进行中：延续“立即同步”主文案，但进入加载态并禁用点击。
- 按钮改成单行布局，主文案左侧、数字状态右侧，整体高度压到与“龙虎榜”一致，仅在宽度上略放大。
- 按钮副文案显示 `(已同步数量/总数)`，例如 `(1/5003)`；若总数暂时不可得，降级为 `(--/--)`。
- 状态判定优先复用现有 `screeningSyncStatus.syncedToLatestStocks` 与 `screeningSyncStatus.marketStockCount`。若同步取消或中断，则前端主动刷新一次 `GetScreeningSyncStatus()`，让按钮即时回到“部分完成”态，而不是停留在初始态。
- 断点续传仍由后端现有 checkpoint 机制负责；前端只负责在按钮文案、进度数字和再次发起同步时保持连续体验。

### 通用同步弹窗

- 从现有筛选确认弹窗中抽出同步通用骨架，支持两种模式：
  - `screening`: 保留“筛选条件”“测试范围”“开始同步并筛选”。
  - `sync-only`: 只展示同步相关信息和“开始同步”按钮，不触发筛选流程。
- 两种模式共用以下区域：
  - 标题与说明
  - 当前同步策略
  - 当前数据状态
  - 同步进度
  - 查看同步设置
  - 取消同步
- 弹窗头部标题与说明文案采用稳定的左对齐布局，避免视觉居中。

### 文案与设置调整

- `SettingsDialog` 左侧选项卡移除“软件更新”。
- AI 筛选工作区主按钮改成两态：
  - 默认显示“开始筛选”
  - 当当前结果来自历史筛选，且输入框内容与该历史记录的 prompt 快照完全一致时，显示“根据历史筛选方式重新筛选”
- 只有在上述历史未改动场景下，点击才直接复用历史 SQL；用户一旦修改输入框文字，就回到重新生成 SQL 的普通 AI 筛选路径。

## 影响范围

- `jcp/frontend/src/App.tsx`
- `jcp/frontend/src/components/SettingsDialog.tsx`
- `jcp/frontend/src/components/ScreeningWorkspace.tsx`
- `jcp/frontend/src/utils/screeningSync.ts`
- 可能新增同步弹窗或状态辅助文件

## 验证

1. 顶部栏出现同步状态按钮，并在三种同步状态下显示正确文案、颜色和禁用态。
2. 部分同步后取消时，按钮会即时显示当前已同步数量，并保留可点击的续传状态。
3. 点击顶部按钮会打开“仅同步”弹窗，不触发筛选。
4. AI 筛选入口仍打开“同步后筛选”弹窗。
5. 两类弹窗头部文案区域左对齐。
6. 设置面板不再显示“软件更新”。
7. 历史筛选结果在未改输入时显示“根据历史筛选方式重新筛选”，改过输入后切回“开始筛选”。
