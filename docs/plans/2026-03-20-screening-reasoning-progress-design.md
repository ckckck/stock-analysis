# AI 筛选连续思考摘要设计

## 背景

当前 AI 筛选虽然已经支持在 `generate_sql` 阶段实时显示 SQL 草稿，但用户在此之前仍需要等待较长时间，体验上像“卡住”了一样。

用户希望更早看到 AI 当前在做什么，但不希望把模型原始思维链直接暴露出来。

## 目标

1. 在 SQL 草稿出现前，就让用户看到 AI 的连续“思考中”反馈。
2. 不展示模型厂商原始 reasoning token 或完整思维链。
3. AI 筛选有结果时默认选中第一只；无结果时左侧明确显示“没有符合条件的结果”。
4. 左侧结果区的页签切换按钮移动到标题和统计文案上方。
5. 保持现有 SQL 生成、校验和执行流程不变，避免影响历史筛选、SQL 查看和结果落库。

## 方案

### 双阶段流式反馈

- 在现有 `prepare -> generate_sql -> validate_sql -> execute_query -> store_results -> completed` 基础上，新增 `reasoning` 阶段。
- `reasoning` 阶段改成连续自然语言流式说明，像“边思考边说明”的摘要，而不是分条短句。
- `generate_sql` 阶段继续沿用现有 SQL 草稿实时输出。

### 后端实现方式

- 保持“思考摘要”和“SQL 生成”分成两次独立模型调用：
  - 第一次调用：生成一段连续自然语言说明，强调不要输出 SQL、不要输出原始推理链、不要用列表格式。
  - 第二次调用：沿用现有 SQL 生成提示词和 SQL 流式展示逻辑。
- 这样做有两个好处：
  - 不需要解析混合输出，避免“思考文本 + SQL”粘在一起。
  - 前端可以更明确地区分“思考中”和“生成 SQL”两个阶段。

### 事件模型

- 扩展 `ScreeningQueryProgress`，但继续复用现有 `streamingText` 字段。
- `currentStage == "reasoning"` 时，`streamingText` 表示连续思考摘要文本。
- `currentStage == "generate_sql"` 时，`streamingText` 表示 SQL 草稿。
- 保持当前 Wails 事件名 `screening:query:progress` 不变，避免额外接线成本。

### 前端展示

- 在现有进度卡中复用同一个流式文本区域：
  - `reasoning` 阶段标题显示为“AI 正在思考”
  - `generate_sql` 阶段标题显示为“AI 当前输出”
- `reasoning` 阶段的文案是逐步追加的一段说明文字，强调“边思考边说明”的观感，而不是列表式步骤。
- `generate_sql` 阶段继续显示当前 SQL 草稿。

### 结果区交互

- 左侧结果区头部改成两层：
  - 第一层先显示“当前筛选结果 / 历史筛选结果”切换按钮。
  - 第二层再显示标题和数量说明。
- 当筛选结果非空时，维持当前行为并自动选中第一只股票。
- 当筛选结果为空时，左侧结果列表明确显示“没有符合条件的结果”，而不是泛化提示文案。

## 不做的事情

- 不展示模型原始 chain-of-thought。
- 不引入新的持久化表保存思考过程。
- 不把思考文本写入历史筛选结果。
- 不修改历史结果重跑逻辑；历史重跑仍直接基于 SQL 重跑。

## 风险与处理

### 额外一次模型调用

- 会增加一次 LLM 请求成本，但这次请求只要求输出几条很短的步骤，成本可控。
- 用户能更早看到反馈，体感会显著好于长时间空等。

### 模型输出不稳定

- 通过单独的 reasoning 提示词约束输出格式，只允许连续自然语言摘要。
- 服务端对空白 chunk、超长文本和异常输出做截断或忽略处理。

### UI 误导为“真实推理链”

- 前端文案明确叫“思考中”，不叫“完整推理”。
- 只展示可控摘要，不展示底层 reasoning token。

## 验证

- Go 测试覆盖：
  - `reasoning` 阶段事件会先于 `generate_sql`
  - `reasoning` 阶段支持流式增量
  - reasoning prompt 要求连续自然语言摘要，不再要求分条输出
  - 不支持 reasoning 流式的生成器仍可回退
- 前端验证：
  - `npm --prefix frontend run build`
  - 检查 `reasoning` 与 `generate_sql` 两种展示标题与内容切换是否正确
  - 检查页签已移动到标题上方，空结果时左侧显示“没有符合条件的结果”
