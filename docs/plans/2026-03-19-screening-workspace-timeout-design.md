# AI 筛选工作区与 SQL 超时设计

## 背景

当前 `AI 筛选` 左下工作区仍保留一段重复说明卡，顶部还有“AI 筛选工作区”标题区，信息密度低，且与左上结果区重复。另一方面，SQL 生成超时目前复用 AI Provider 的 `timeout`，用户很难在 `AI筛选` 场景里直接找到并理解这个配置。

## 目标

1. 精简左下工作区，只保留自然语言输入、结果条数选择和执行按钮。
2. 去掉“AI 筛选工作区”标题区，将输入区提示文案收敛成 `输入自然语言，描述筛选方式`。
3. 在 `AI筛选` 设置里新增独立的 `SQL 超时` 配置，并让筛选 SQL 生成优先使用它。

## 方案对比

### 方案 A：沿用 Provider `timeout`

- 优点：代码改动最少。
- 缺点：语义混乱，用户需要在模型设置里推断“这其实是 AI 筛选 SQL 超时”。

### 方案 B：在 `AI筛选` 中新增独立 `SQL 超时`

- 优点：配置位置符合心智模型，后续可以独立调优，不影响其它 AI 请求。
- 缺点：需要补配置字段默认值、迁移和读取优先级。

## 选型

采用方案 B。

## 设计细节

### 工作区 UI

- 删除左下说明卡。
- 删除顶部标题区和数据库图标。
- 将原“筛选条件”标题替换为单行副标题：`输入自然语言，描述筛选方式`。
- 保持当前文本框、结果条数选择和执行按钮布局不变，使工作区自然贴近左下区域。

### 超时配置

- 在 `models.ScreeningConfig` 中新增 `SQLTimeoutSeconds`。
- 默认值使用 `45` 秒。
- 旧配置加载时，如果该字段缺失或为 `0`，回填默认值。
- `ScreeningQueryService.resolveSQLGenerationTimeout()` 优先读取 `config.Screening.SQLTimeoutSeconds`，只有为空时才回退到现有 AI Provider 的 `timeout`。

### 设置页

- 在 `AI筛选` 设置页加入 `SQL 超时` 选择项。
- 选项先做成固定值：`45 / 60 / 90 / 120 / 180` 秒，避免自由输入带来的无效值。
- 确认弹框里的 `SQL 超时` 摘要改为显示 `AI筛选` 配置，而不是从默认 Provider 推断。

### 验证

- 后端测试覆盖：`AI筛选` 超时字段优先级高于 Provider `timeout`。
- 前端验证通过 `npm run build`。
- 全量验证通过 `go test ./...` 与 `wails build`。
