# 统一日志优化设计（行情 / AI / 空间治理）

## 背景

当前 `jcp` 已具备基础文件日志能力，日志文件按天写入 `logs/YYYY-MM-DD.log`。但日志覆盖存在明显不均衡：

- AI 筛选链路已有较完整的进度与存储日志。
- 会议 / OpenClaw 链路已有一定的阶段性日志。
- 普通行情链路（实时行情、K 线、盘口）日志明显不足。
- 前端大量关键失败只停留在 `console.warn/error`，未稳定落到文件日志。
- Go 桥接层有多处直接吞错，导致界面症状与文件日志脱节。
- 当前默认全局日志级别为 `DEBUG`，若继续补高频日志而不做治理，长期会带来空间与噪声问题。

本设计目标不是“把所有细节都打出来”，而是建立一套统一、可控、可长期运行的日志体系。

## 目标

1. 让主要业务链路都能回答三个问题：
   - 请求有没有发出？
   - 失败发生在哪一层？
   - 为什么失败？
2. 让前端、Go 桥接层、服务层日志可以串起来，而不是各自为政。
3. 让日志长期可用，不因高频 debug 淹没关键信息或占满磁盘。
4. 第一阶段优先解决 Windows 现场排查最常见的“界面有症状、文件日志没证据”的问题。

## 范围

### 第一阶段纳入范围

- 普通行情链路：实时行情、K 线、盘口
- 前端文件日志桥接：warning / error / 关键用户动作
- 会议入口摘要日志
- 日志空间治理基础能力

### 暂不在第一阶段完成

- 全量结构化 JSON 日志改造
- 历史日志上传/云端集中采集
- 所有模块的 request id 全量贯通
- 图表交互调试日志的完整重构

## 现状问题

### 1. 普通行情链路缺少关键日志

- `GetKLineData`、`GetStockRealTimeData`、`GetOrderBook` 的 Go 桥接层存在直接吞错的情况。
- `MarketService` 在请求失败时大多只返回 error，不主动落业务日志。
- `MarketDataPusher` 更偏向推送框架状态，缺少订阅 symbol、周期、结果条数等业务上下文。

### 2. 前端 warning / error 未稳定落盘

- 当前 `debugAppEvent()` 会调用 `LogFrontendDebug()`，可落入文件日志。
- 当前 `warnAppEvent()` 仅 `console.warn + LogWarning`，没有像 `debugAppEvent()` 那样稳定桥接到文件日志。
- 很多关键失败仍使用裸 `console.warn` / `console.error`，现场日志不可见。

### 3. 高价值链路缺少统一摘要

- 会议入口缺少对 `stockCode`、模式、消息长度的请求摘要。
- OpenClaw 缺少统一 access summary。
- 快讯 / 指数 / 新闻等链路失败时不易快速区分“空数据”与“请求失败”。

### 4. 当前没有长期空间治理能力

- 日志按天切分，但无自动清理。
- 无历史压缩。
- 无总量上限。
- 当前默认全局 `DEBUG` 不适合长期运行。

## 统一日志设计原则

### 1. 分层记录

日志分为四层：

1. 用户动作层：谁触发了什么。
2. 桥接层：前端到 Go 的关键接口调用。
3. 服务层：真正执行 HTTP / DB / AI / 推送的逻辑。
4. 诊断层：高频 debug，仅在需要时临时开启。

### 2. 失败详细、成功摘要

- 失败日志要尽量给出参数摘要、阶段、错误原因。
- 成功日志只打摘要，不打印高体积 payload。

### 3. 高频日志必须可控

- 高频成功日志默认不打细节。
- 高频 warning / error 必须支持限频或折叠。
- 流式 / 交互类 debug 默认不长期常开。

### 4. 现场排查优先

日志体系必须优先服务 Windows 现场排查，而不是只服务开发期阅读源码。

## 第一阶段日志补齐方案

### A. K 线链路

#### 前端

在切换股票或周期时记录：

- `symbol`
- `period`
- `days`
- `requestId`
- `attempt`

在以下情形记录 warning / error 并落盘：

- 返回空数组
- 请求抛错
- 多次重试后仍失败

#### Go 桥接层

`App.GetKLineData` 增加：

- 调用开始摘要
- 调用失败日志

不再静默吞掉 error。

#### 服务层

`MarketService.fetchKLineData` 增加：

- 请求开始（debug 级）
- HTTP 非 200（warn）
- 解析失败（warn）
- 成功返回条数（debug 或 info，视级别策略决定）

### B. 实时行情链路

`GetStockRealTimeData` 与对应服务层增加：

- 调用股票数量摘要
- 请求失败摘要
- 返回条数摘要

### C. 盘口链路

`GetOrderBook` 与对应服务层增加：

- symbol 摘要
- 请求失败摘要
- 返回买卖档数量摘要

### D. 前端 warning / error 桥接

新增统一原则：

- `debugAppEvent()`：调试级落盘
- `warnAppEvent()`：warning 级落盘
- 新增 `errorAppEvent()`：error 级落盘

不再允许关键业务失败只停留在裸 `console.warn/error`。

### E. 会议入口摘要

`SendMeetingMessage` 增加入口摘要日志，记录：

- `stockCode`
- 是否指定专家
- 请求文本长度
- 会话是否存在

第一阶段只补入口摘要，不立即重构完整会议 request id。

## 统一字段建议

第一阶段先在日志消息中统一这些核心字段：

- `module`
- `action`
- `symbol` / `stockCode`
- `period`
- `requestId`（先从 K 线开始）
- `attempt`
- `resultLen`
- `durationMs`
- `err`

第一阶段不强制改造成 JSON 行日志，但消息文本要尽量保持稳定结构，方便 grep。

## 空间治理方案

### 1. 默认级别策略

- 长期默认：全局 `INFO`
- 排障临时：按模块开 `DEBUG`
- 高频模块如 `pusher` 默认不长期跑 `DEBUG`

目标是不再长期全局 `DEBUG`。

### 2. 模块级别控制

后续支持至少这些模块独立调级：

- `frontend`
- `market`
- `pusher`
- `screening.sync`
- `screening.query`
- `meeting`
- `openclaw`

第一阶段若不做完整模块配置，也要至少把全局长期级别从 `DEBUG` 收回。

### 3. 高频限频

对同 key 重复日志做限频：

- 首次必打
- 时间窗口内同 key 折叠
- 下一次输出时附带 `suppressedCount`

适用场景：

- 某股票 K 线连续空数据
- 某接口持续超时
- 某推送循环重复失败

### 4. 历史保留策略

建议默认：

- 最近 7 天保留原始日志
- 8-30 天压缩为 `.gz`
- 超过 30 天自动删除

如需更保守，可改为 14/60 天，但第一阶段建议先实现 7/30。

### 5. 总量上限

给 `logs/` 目录增加总量上限，例如 300MB。

超过上限时，优先删除最老的压缩日志，再删除最老的原始日志。

## 不建议长期保留为 debug 的日志

- 图表缩放 / 键盘交互细节
- 高频推送每次的详细 payload
- AI 流式 delta 逐段输出
- 每轮轮询的全量响应正文

这些日志可保留能力，但默认不长期常开。

## 长期建议保留的日志

- 用户关键动作摘要
- 桥接层关键接口摘要
- 空数据 warning
- fallback warning
- 请求失败 / 超时 / 解析失败
- AI 任务开始 / 结束 / 失败摘要
- OpenClaw access summary

## 实施顺序

### Phase 1

1. 前端 warning / error 落盘能力补齐
2. K 线链路补日志
3. 实时行情与盘口链路补日志
4. 会议入口摘要补齐

### Phase 2

1. 默认日志级别治理
2. 日志保留 / 压缩 / 删除策略
3. 总量上限控制

### Phase 3

1. 模块级别控制
2. 高频限频
3. request id 进一步贯通多链路

## 风险与权衡

### 风险

- 若一次性补太多 debug，短期日志量会显著增加。
- 若字段格式不统一，后续 grep 与排障收益有限。
- 若默认级别不调整，即使加了保留策略，噪声仍然会高。

### 权衡

- 第一阶段优先补“失败可见性”，不追求完整结构化。
- 空间治理与日志补齐同步设计，但实现可以分 Phase 1 / Phase 2 落地。
- 先做最影响 Windows 现场排查的普通行情链路，再扩展到更广范围。

## 验收标准

满足以下标准即可认为第一阶段达标：

1. 当普通 K 线加载失败时，Windows 日志中能看到：
   - 前端是否发起了请求
   - Go 桥接层是否收到请求
   - 服务层失败原因或空数据摘要
2. 当实时行情或盘口失败时，日志中不再只有界面症状，没有调用证据。
3. 前端关键 warning / error 不再只存在于浏览器控制台。
4. 日志默认运行模式不会因为新增日志而明显失控。
