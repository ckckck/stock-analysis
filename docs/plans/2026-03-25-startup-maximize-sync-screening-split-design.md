# 启动最大化与同步/筛选拆分设计

## 背景

当前桌面应用启动后仅以固定窗口尺寸打开，不会自动最大化到工作区。AI 筛选确认弹框在 `screening` 模式下也只有一个“开始同步并筛选”按钮，导致“同步”和“筛选”两个动作无法分开执行。

## 目标

1. 应用启动时自动最大化到桌面工作区，不进入系统级全屏。
2. `screening` 模式确认弹框底部拆成两个动作：
   - 左侧次级强调按钮：`开始同步`
   - 右侧主按钮：`开始筛选`
3. 左按钮只负责同步，右按钮直接基于当前本地数据库执行筛选，不额外触发同步。
4. `sync-only` 模式保持原有单按钮行为不变。

## 方案

### 启动窗口

将 Wails 应用窗口配置改为 `WindowStartState = options.Maximised`。这样启动后窗口会自动最大化到工作区，同时保留当前无边框窗口和手动还原/最大化交互。

### 筛选确认弹框

保留现有 `SyncActionDialog` 组件，但仅在 `mode === "screening"` 且未进入同步进度态时渲染双按钮：

1. `开始同步`
   - 沿用当前手动同步逻辑。
   - 同步完成后只刷新状态，不自动继续 AI 筛选。
2. `开始筛选`
   - 关闭弹框。
   - 直接调用现有筛选执行链。
   - 历史 SQL 复用仍按原有条件判断。

### 文案

`screening` 模式标题和说明从“确认同步后开始筛选”改成“确认后执行同步或筛选”，明确右侧按钮不会自动同步，只会基于当前已同步数据执行筛选。

## 风险与约束

1. 用户可以在数据不是最新时直接点“开始筛选”，这是目标行为，不再做强制拦截。
2. 仍保留当前数据状态卡，继续向用户暴露“是否已同步到最新”的状态。
3. 只改 `screening` 模式，不改顶部栏 `sync-only` 弹框，避免引入额外回归面。

## 验证

1. Go 测试覆盖窗口启动配置函数返回 `options.Maximised`。
2. 前端 `vitest` 覆盖 `screening` 模式弹框文案和双按钮文案。
3. 至少手动验证：
   - 启动即最大化。
   - `开始同步` 不触发筛选。
   - `开始筛选` 不触发同步。
